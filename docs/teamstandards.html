Checklist : Team Development Standards

  • Small classes with few methods are far better than large classes with lots of methods.
  • Small methods are far better than large methods.
  • Use meaningful names for classes, methods and variables.
  • Use javadoc comments on classes and public methods. If you feel you need to add comments to private methods or within the body of a method, consider refactoring. Comments are not necessary for POJO accessors/mutators (getters/setters).
  • Keep comments short, simple, and to the point.
  • When you update code, update any relevant comments
  • Anything that can be unit tested should be pushed/merged with automated unit test(s).
  • Test code should be developed with the same attention to detail and care as any other code.
  • If you have to refactor code in order to test it, by all means refactor the code!
  • Abide by the Single Responsibility Principle. Every class and method you write should have one responsibility.
  • Never push or merge code that has not been thoroughly tested. Your code is your reputation.
  • Use checkstyle with the included configuration, and validate your code with it before pushing/merging
  • Use findbugs and validate your code with it before pushing/merging.
  • Check your code out in Sonar before pushing/merging, and make sure it does not introduce any rule violations or other quality issues.
  • Cutting & pasting code is evil. Write small, reusable methods and reuse them where possible!
  • Avoid reinventing the wheel. Gain some familiarity with existing code as well as the project dependencies, and consider using existing code before writing redundant code. For example, familiarize yourself with commons-lang and commons-beanutils.
  • Most Importantly - Follow the Golden Rule. Treat others the way you'd like to be treated. Remember that your code will likely be maintained and extended by others. Don't make it hard for them.
